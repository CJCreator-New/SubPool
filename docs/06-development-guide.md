# SubPool Software Development Guide

## 1. Environment Setup

### Prerequisites
* Node.js (v18 or higher recommended)
* pnpm (or npm, though `pnpm-lock.yaml` exists in the repo)
* Git
* Supabase CLI (if running the backend locally)

### Installation
1. Clone the repository: `git clone <repo_url> && cd SubPool`
2. Install dependencies: `npm install` (or `pnpm install`)
3. Create `.env` based on `.env.example`:
   ```bash
   cp .env.example .env
   # Ensure you populate VITE_SUPABASE_URL and VITE_SUPABASE_ANON_KEY 
   ```

### Running Locally
* Start the Vite development server:
  ```bash
  npm run dev
  ```
  This will serve the application at `http://localhost:5173`.

## 2. Coding Standards & Conventions

### TypeScript
* **Strict Mode:** Always use strict typing. Avoid `any`. Use generics and existing DB types generated by Supabase (`src/types/supabase.ts` or similar).

### React & UI
* **Components:** Use functional components with hooks. Destructure props.
* **Styling:** Use Tailwind CSS exclusively. Do not write vanilla CSS unless absolutely necessary (e.g., in `index.css` for root variables). Use `clsx` and `twMerge` via the `cn()` utility for conditional class merging.
* **State Management:** Keep state local when possible. Use React Context strictly for global, non-frequent state. Use Supabase hooks / `react-query` for server state.

### File Structure
* `/src/components`: Reusable UI components (buttons, modals, Radix primitives).
* `/src/pages`: Top-level route components.
* `/src/lib` or `/src/utils`: Helper functions, custom hooks, and shared logic (`utils.ts`, supabase clients).
* `/supabase/migrations`: Version-controlled SQL files.

## 3. Version Control Workflow
1. **Branch Naming:**
   * Features: `feat/[feature-name]`
   * Bug Fixes: `fix/[bug-name]`
   * Refactors: `refactor/[task]`
2. **Commits:** Conventional Commits format is preferred (e.g., `feat: added pool messaging ui`).
3. **PR Process:**
   * Create a PR against the `main` or development branch.
   * Ensure tests pass (`npm run test:run` and `npm run test:e2e`).
   * Require at least one review.

## 4. Contributing to the Database
* **Migrations:** Do not modify the database manually via the Supabase dashboard in production.
* Create a new migration file: `npx supabase migration new [migration_name]`
* Write idempotent SQL statements if possible.
* Apply locally: `npx supabase db push` (or `start` + `db reset`).
